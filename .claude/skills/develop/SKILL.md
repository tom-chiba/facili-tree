# 実装フロー

`$ARGUMENTS` のイシューを下記 7 ステップで進める。

フローは Task ツール（`TaskCreate` / `TaskUpdate` / `TaskList`）で管理する。着手時に 7 ステップを `TaskCreate` で登録し、開始するステップを in_progress にして本スキルを読み直し（やること・差し戻しを確認）、やることを終えたら completed にして次へ進む。in_progress は常に 1 つだけにし、Task を飛ばして先のステップに進まない。

## 差し戻し

問題の根本が前のステップにあると分かったら、そのステップまで戻り、戻り先以降を pending に戻して再開する。どのステップからでも、それより前のほぼすべてが戻り先になり得る。「どこで誤ったか」で戻り先を選ぶ。例:

- 仕様の取り違え・受入基準の見落とし → 1
- コードベースの理解不足・想定外の依存 → 2
- 計画の穴・検証手段の誤り → 3
- 実装の不備 → 4

## ステップ

### 1. 把握

GitHub Issue 番号なら gh CLI（`gh issue view`）で取得、課題文・仕様ならそのまま読む。画像・参照ファイル・URL 等のリッチコンテキストを取り込む（CLI/MCP で届くのはテキストだけで画像・動画は中身が見えず、参照ごと欠落して気づけないこともある。リッチコンテキストの欠落を疑い、必要なら添付を依頼）。曖昧・情報不足なら質問して仕様・受入基準を固める。

### 2. 探索

関連コード・データフロー・既存パターンを変更せず理解する。広い調査は 複数の Explore サブエージェントに委譲し要約だけ受け取る。

### 3. 計画

変更ファイル・内容・影響範囲・検証手段を列挙する（検証手段はここで決める）。一文で説明できる小修正は簡略化、アプローチが不確実・複数ファイル・不慣れなコードなら計画を作成する。

### 4. 実装

イシュー用ブランチで作業する（わかりやすい名前で切る。base は main、関連する親 Issue があればそのブランチ）。未作成なら切り、既にあれば（差し戻しで戻った場合など）そのまま使う。計画に従い、小さな対応ごとにこまめに commit する。

### 5. 検証

計画で決めた検証を実行し、証拠（コマンドと戻り値・テスト出力・スクショ）を確認する。動作確認と UI 変更は playwright-cli で実画面確認。

### 6. レビュー

base ↔ ブランチの全差分を、観点ごとに複数の general-purpose サブエージェントへ並列委譲する（専用レビューエージェントは使わず純粋な観点レビュー。観点: 正確性 / 全要件・エッジケースのテスト / 設計（責務分割・抽象化・型設計・共通化）/ スコープ外の変更）。スタイル好み・過剰防御は追わない。

各指摘には該当行と根拠を必須とし、`git diff`・`git blame` で今回差分起因か既存かを確かめさせる。集約時、裏付けのない指摘・今回起因でない既存事象は棄却する。

指摘はここで直さず（軽微・一行でも例外なし）、`差し戻し` に従い根本ステップへ Task を戻して着手する。

新たな指摘ゼロのクリーンなレビューが 1 回通るまで completed にしない。指摘が出るたび 差し戻し → 修正 → Step 5 → 全差分・全観点で再レビュー、を収束まで繰り返す。

### 7. 仕上げ

仕様やコードベースを知らない人向けに変更内容を簡潔に報告。main へは直 push せず、PR を出してレビューする方針。
